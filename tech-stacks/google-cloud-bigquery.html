Google Cloud BigQuery

In our experience, many Data Mesh implementations rely on Google Cloud Platform as a common infrastructure, at least for analytical data. It is a highly integrated platform. Everything is available as developer-friendly self-service. The on-demand query performance is remarkable.

Every domain team typically gets their own GCP project(s) under an organization resource. There is a separate project for each stage, such as test and production.

BigQuery is the central component for storing analytical data. BigQuery is a columnar data store and can perform efficient JOIN operations with large data sets. It supports access to files stored in Google Cloud Storage as external tables, so it can also be used as a data lake, if required. Manual data is easily edited with Google Sheets.

BigQuery supports both, batch ingestion and streaming ingestion. When the operational system architecture relies on Apache Kafka, then streaming ingestion Kafka Connect Google BigQuery Sink Connector. When the operational system architecture does not rely on Apache Kafka, then streaming ingestion often is done with Google Pub/Sub and a simple Cloud Function to forward events to BigQuery.

For analytics, BigQuery supports SQL queries with a number of available user-defined functions. For basic visualization and report generation, Google Data Studio is well-integrated and can be used for free. A good integration also exists for Google Sheets and Colab, an online Jupyter Notebook service. More advanced analytics is done with Looker, which was acquired by Google in 2019.

Data Products are usually simple BigQuery tables or views in a marked dataset. The access to these data is managed by the domain teams with Google IAM roles. Google has an integrated Data Catalog, which is pretty cluttered. It can be used to document and tag datasets as data products, but we see that many users still stick on a Wiki page with available data products listed.
